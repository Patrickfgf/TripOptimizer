TripOptimizer

Otimizador da ordem mais barata de uma viagem multi-cidade: dado um conjunto de destinos, dias em cada, aeroporto de saída/retorno e uma janela de datas flexível, computa a sequência de voos de menor custo total — porque a ordem muda as datas e as datas mudam o preço.

Python 3.11+ · FastAPI DuckDB · Parquet React 18 · TS · Vite EUR · Europa (sample) Projeto de portfólio (DA · DS · DE · Full-Stack)

Visão geral

Ao planejar uma viagem por vários países, o plano mais barato raramente é a ordem que você imagina primeiro. A ordem das cidades muda a data de cada voo (os dias acumulam), e a data muda o preço. O viajante acaba re-checando manualmente cada permutação da rota e cada variação de data — exatamente a dor vivida no Erasmus em Portugal (15 países).

Input

cities[], days_per_city, origin_airport, return_airport, start_date, flex_days (±N).

Output

Itinerário mais barato = melhor (ordem das cidades, offset de data, voos por leg, custo total) + lista ranqueada de alternativas.

Guardrails: máx. 8 cidades (n! tratável / Held-Karp 2ⁿ·n² exato), flex_days default ±3 (cap ±7), moeda fixa em EUR. Voos apenas no MVP.

Arquitetura

Monorepo, core Python puro + borda React. I/O nas bordas; o core não conhece HTTP nem disco.

tripoptimizer/
├─ backend/
│  ├─ core/
│  │  ├─ optimizer/   # busca da rota mais barata (puro, sem I/O)
│  │  ├─ fares/       # FareProvider: Travelpayouts | Cached | Synthetic | Fallback
│  │  └─ graph/       # aeroportos/IATA (OurAirports) + haversine
│  ├─ api/            # FastAPI: /optimize, /airports, /health
│  ├─ ingestion/      # offline: Travelpayouts → snapshot normalizado (CLI idempotente)
│  └─ tests/          # pytest (oráculo DP-vs-brute, contract tests, integração httpx)
├─ frontend/          # React 18 + TS + Vite + Tailwind + shadcn/ui   ← Plan 4
├─ data/              # snapshot committed (Parquet) + grafo de aeroportos
└─ docs/              # specs + este guia interno (HTML) + README (Plan 5)

Decisões (why-X-not-Y)

DecisãoEscolhaPor que não a alternativa
StackPython core (FastAPI + DuckDB) + React/TSCobre DE+DS+DA+Full-Stack num projeto só. Descartado TS/Supabase-Edge puro (sub-usa o stack de dados) e Streamlit-only (perde o sinal full-stack).
Fonte de faresTravelpayouts (cache real) + fallback sintéticoToken grátis e instantâneo; tem cheapest-by-date. Amadeus está descomissionando o self-service (jul/2026); Skyscanner é partner-gated; Kiwi fechou o self-service; Duffel só dá dado sintético no sandbox.
ReprodutibilidadeSnapshot committed, nunca live-per-requestBusca combinatória (~N² lookups) queima qualquer quota e quebra a reprodutibilidade. Snapshot + cache por (origin,dest,date) resolve.
AlgoritmoBrute-force (MVP) + Held-Karp (DP sobre subconjuntos)Dias fixos ⇒ a data do k-ésimo leg depende só do conjunto já visitado ⇒ DP 2ⁿ·n² exato. Brute-force é também o oráculo de teste.
Frontend (Plan 4)Página única + estado na URL · Warm Bento · mapa SVG keylessRouter/wizard são scaffolding demais p/ fluxo único. Dark mode é o look "esperado". Mapa por tiles quebra o "roda sem API key".

Camada de dados

Grão da linha de fare: 1 linha = origin × destination × date × fare. Persistido em Parquet (tipado, colunar), consultado via DuckDB.

raw → curated

Ingestão idempotente: raw read-only → snapshot curado. Re-rodar = mesmo resultado.

Proveniência por leg

Leg.source propaga do provider vencedor até a API; data_source agrega (cached/synthetic/mixed).

Fallback gracioso

Célula (A,B,date) faltante → fare sintético determinístico. O demo nunca retorna vazio.
